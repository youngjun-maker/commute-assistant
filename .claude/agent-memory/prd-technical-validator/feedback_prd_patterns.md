PRD 기술 검증을 수행하면서 발견된 반복 패턴들을 기록.

**패턴 1: user_id 존재하나 Auth 설계 누락**
스키마에 user_id가 있는데 인증 방식(로그인, 익명, 기기 ID 등)이 PRD에 없는 경우가 빈번. 1인 앱에서 특히 자주 발생.

**패턴 2: CORS 아키텍처 미명시**
외부 API(카카오, 공공데이터포털 등)를 사용하는 PRD에서 클라이언트-서버 경계가 명시되지 않아 CORS 문제가 구현 중 블로킹으로 발생.

**패턴 3: Fallback 데이터 출처 미정의**
"API 장애 시 평균값으로 대체" 류의 fallback 정책이 명시되나, 그 평균값이 어디에 저장되는지 스키마에 없는 경우.

**패턴 4: 성능 목표 vs 실시간 데이터 충돌**
LCP 3초 이하 + 앱 실행 시 실시간 API 데이터 즉시 표시는 동시 달성이 어려움. Skeleton UI + 지연 로딩 전략 명시가 필요.

**패턴 5: 지역 범위 미정의 (한국 공공 API)**
한국 공공데이터포털 API는 지역별로 API가 분리되어 있어 지원 지역 명시가 필수. 서울 버스 API로 경기도 대응 불가.

**패턴 6: next-pwa v15 호환성 이슈**
Next.js v15 App Router 프로젝트에서 next-pwa 사용 권장 시 호환성 경고 필요. Serwist가 현재 더 안정적.

**패턴 7: 외부 API ID 체계 불일치 (다중 API 연계)**
두 개 이상의 외부 API를 연계할 때, A API에서 얻은 ID를 B API에 그대로 사용하는 설계는 ID 체계가 동일하다는 가정이 필요. 이 가정은 반드시 사전 검증되어야 하며 PRD에 "Week N 검증 항목"으로 미루는 것은 아키텍처 전체의 블로킹 리스크가 됨. (발견 사례: ODsay startID ↔ 공공데이터포털 stId)

**패턴 8: 출발지 좌표의 DB 누락**
경로 탐색 API(ODsay, 카카오 등)를 사용하는 PRD에서 목적지 좌표는 명시되나 출발지(홈 주소) 좌표의 저장 위치가 스키마에 없는 경우가 발생. 경로 탐색 API의 필수 파라미터이므로 DB 스키마 설계 시 반드시 체크.

**패턴 9: 교통 수단 타입 컬럼 부재 (버스/지하철 분기)**
버스와 지하철 실시간 API를 모두 지원하는 설계에서, 어느 API를 호출해야 하는지 결정하기 위한 `traffic_type` 컬럼이 DB에 없는 경우. ODsay 응답의 `trafficType` 값을 반드시 저장해야 함.

**패턴 10: 외부 API 응답 필드 경로의 코드 예시 오류**
PRD에 코드 예시가 포함된 경우, 실제 API 응답 구조와 다른 필드 경로를 참조하는 오류가 발생할 수 있음. 특히 중첩 배열(lane[], subPath[])에서 같은 이름의 필드가 다른 레벨에 있는 경우 혼동됨. (발견 사례: ODsay 버스 arsID가 lane[0] 레벨이 아닌 subPath[n] 레벨에 있을 가능성) — Week 1에서 실제 응답 JSON 로깅으로 반드시 확인.

**패턴 11: Anonymous Auth 세션 만료와 폴링의 충돌**
Supabase Anonymous Auth 기본 세션 1시간 만료 → 30초 이상의 반복 폴링이 있는 앱에서 세션 만료 에러 처리 로직이 없으면 폴링이 401 에러 루프에 빠짐. React Query onError 또는 Supabase auth.onAuthStateChange를 통한 자동 갱신 구현 필요.

**패턴 12: Route Handler URL 파라미터명 ↔ 외부 API 파라미터명 불일치**
Route Handler가 받는 쿼리 파라미터명(예: `arsId`)과 외부 API 호출에 사용하는 파라미터명(예: `stationID`)이 다른 경우, PRD에 변환 매핑이 명시되지 않으면 구현자가 그대로 전달하여 외부 API 호출 실패 가능. (발견 사례: ODsay 버스 실시간 API에서 Route Handler `arsId` → ODsay `stationID` 매핑 누락)

**패턴 13: 타임존 미명시 (INT/TIME 컬럼의 KST vs UTC 혼용)**
Supabase는 기본 UTC를 사용하나, 사용자 입력은 KST 기준. `TIME` 타입 컬럼이나 시각 비교 INT 컬럼(출근 시작/종료 시각)의 저장 및 비교 기준이 명시되지 않으면 9시간 오차 발생. "모든 시각은 KST 기준, 클라이언트 로컬 시간으로 비교" 한 문장 명시로 해결 가능.

**패턴 14: Supabase 무료 티어 7일 비활성 자동 정지 미언급**
Supabase 무료 티어는 7일간 API 호출이 없으면 프로젝트가 자동 정지됨. 재가동 시 수십 초 소요 → LCP 성능 목표 위반. 1인 앱 또는 소규모 앱 PRD에서 반복 누락. UptimeRobot 등 무료 uptime ping으로 예방 가능.

**패턴 15: PRD 섹션 순서 역전으로 인한 로직 누락 위험**
PRD 편집 과정에서 섹션 헤더와 내용이 분리되거나 순서가 역전되는 경우, 개발자가 해당 섹션이 비어 있다고 판단하여 중요한 구현 로직을 누락할 수 있음. (발견 사례: Mom's Route v1.4 — 4-5 헤더 이후 내용 없음, 실제 내용이 4-6 이후에 위치하여 세션 복구 로직 누락 위험) PRD 최종 검수 시 각 섹션 헤더와 내용의 연속성을 반드시 확인.

**패턴 16: 보조 테이블 캐시 컬럼의 사용 로직 미정의**
메인 테이블(schedules)의 캐시 컬럼 만료 로직은 명세되어 있으나, 보조 테이블(overrides)의 동일 구조 캐시 컬럼 사용 시나리오가 없는 경우. 오버라이드처럼 생명주기가 짧은 레코드에서는 캐시 만료 개념 자체가 불필요할 수 있으며, 이 경우 "감사 로그 전용"임을 명시해야 불필요한 로직 구현을 방지.

**패턴 17: JSONB 캐시 필드의 접근 경로 미명시**
JSONB 컬럼에 외부 API 전체 응답을 저장하고 특정 필드(totalTime 등)를 이후 로직에서 사용하는 경우, 실제 JSONB 내 접근 경로(예: `cache.info.totalTime` vs `cache.totalTime`)가 PRD에 명시되지 않으면 구현 오류 발생. 코드 예시에서 필드 추출 경로를 명시하거나 PostgreSQL JSONB 쿼리 예시를 포함해야 함.

**패턴 18: 최초 진입(신규 사용자) 플로우 미명세**
예외 처리 섹션에서 "스케줄 미등록" 케이스는 처리하면서 "전역 설정(user_settings) 미등록" 케이스를 누락하는 경우가 있음. 경로 탐색 API의 출발지 좌표가 없으면 앱이 동작 불가 상태가 되므로, 최초 진입 시 설정 화면으로 강제 이동하는 플로우를 예외 처리 섹션에 명시해야 함.

**패턴 19: Auth 이벤트 오용 — TOKEN_REFRESHED 시 재인증 호출**
세션 복구 목적으로 onAuthStateChange에서 signInAnonymously()를 재호출할 때, SIGNED_OUT만 트리거로 사용해야 함. TOKEN_REFRESHED는 토큰 갱신 성공 이벤트이므로 이 시점에 재인증을 호출하면 기존 세션이 살아있는 상태에서 중복 호출 발생 → 라이브러리 버전에 따라 새 anonymous 사용자(새 user_id) 생성 가능 → DB 데이터 연결 끊어짐. (발견 사례: Mom's Route v1.4 — 4-5 세션 복구 이벤트 목록에 TOKEN_REFRESHED 포함)

**Why:** 이 패턴들은 개발 Week 1~3 사이에 블로킹 이슈로 발현되며, PRD 단계에서 해결 시 개발 지연을 방지.

**How to apply:** PRD 검증 시 위 패턴들을 체크리스트로 활용. 다중 API 연계 설계에서는 패턴 7, 8, 9, 10, 12, 17을 집중 점검. 세션 기반 인증 + 폴링 조합에서는 패턴 11을 반드시 확인. Supabase 무료 티어 사용 프로젝트에는 패턴 14를 체크. PRD 최종 검수 시 패턴 15(섹션 순서 연속성)를 체크.
